# tsumeshogi-web-solver publish research

## Goal

`https://github.com/koba-e964/tsumeshogi-web-solver` の公開物を、このリポジトリの GitHub Pages 配下 `tsumeshogi-web-solver/` として配信できるようにする。

## Relevant files and modules

### This repository

- `README.md`
  - 配信方針。公開物は `pages/` 直下またはサブディレクトリ配下に置き、`.github/workflows/deploy-pages.yml` で `dist-pages/` に組み立てて Pages へ配信する。
- `.github/workflows/deploy-pages.yml`
  - 既存では `pages/.` と `fund-price-forecast/site/.` を `dist-pages/` にコピーして Pages artifact を作る。
  - 追加コンテンツはここで `dist-pages/<path>/` にコピーすれば配信対象に入る。
- `pages/`
  - ルート公開物。
- `fund-price-forecast/site/`
  - 既存のサブパス配信例。静的成果物を repo に保持して workflow で所定の公開パスへコピーする。

### `tsumeshogi-web-solver` repository

- `README.md`
  - 現在は GitHub Pages 上で単独公開している前提。
- `.github/workflows/gh-pages.yml`
  - `www/dist` を GitHub Pages artifact としてアップロードしている。
  - ビルド手順は `rust-toolchain` + `cargo install wasm-pack` + `npm install` + `npm test` + `npm run build`。
- `Cargo.toml`, `src/lib.rs`
  - Rust/WASM solver 本体。
- `www/package.json`
  - フロントエンド build entry。`npm run build` は `webpack build --mode production`。
- `www/webpack.config.js`
  - 出力先は `www/dist`。
  - `CopyWebpackPlugin` で `index.html` と `favicon.ico` を dist にコピー。
  - `WasmPackPlugin` で Rust crate から wasm bundle を生成する。
- `www/index.html`
  - `./main.js` を読むだけの単純な静的エントリ。相対パス利用なのでサブディレクトリ配信と相性がよい。
- `www/src/whole_player.tsx`
  - UI 本体。
  - `../` へのリンクを持っており、`/tsumeshogi-web-solver/` 配下に置いたときはサイトルートへ戻る導線としてそのまま機能する。
- `www/src/proxy_fetch.ts`
  - 外部 KIF 取得に `https://api.codetabs.com/v1/proxy/` を使用。今回の配信作業ではこの依存先は変えなくてもよい。

## Execution flow and call graph

### Current site deploy flow

1. `main` への push で `.github/workflows/deploy-pages.yml` が走る。
2. workflow が `dist-pages/` を作成する。
3. `pages/.` を `dist-pages/` にコピーする。
4. `fund-price-forecast/site/.` を `dist-pages/fund-price-forecast/` にコピーする。
5. `dist-pages/` 全体を GitHub Pages artifact として upload / deploy する。

### `tsumeshogi-web-solver` build and runtime flow

1. `npm run build` in `www/`
2. webpack build starts from `www/src/index.tsx`
3. `index.tsx` creates a Web Worker from `worker.ts`
4. `worker.ts` imports wasm package from Rust crate output generated by `WasmPackPlugin`
5. build result is emitted to `www/dist/`
6. `www/index.html` loads `./main.js`
7. runtime UI in `whole_player.tsx` mounts shogi player, reads query `sfen`, and dispatches solving to the worker

## Data structures and invariants

- GitHub Pages artifact is assembled, not generated by a framework. Files copied into `dist-pages/<subpath>/` are published as-is.
- `tsumeshogi-web-solver` frontend is static output with relative asset references, so it does not require origin-root deployment.
- Runtime expects generated JS / worker / wasm files to stay together under one directory.
- Existing site convention is "subproject keeps its own assets under a dedicated directory, workflow places them under the final public path".

## Existing architectural patterns

- This repo is a monorepo for multiple public artifacts, not a single build system.
- Per-content directories own their source and/or generated site assets.
- Final publishing is centralized in one Pages workflow.

## Naming conventions

- Public path names mirror directory names, e.g. `fund-price-forecast/site` -> `/fund-price-forecast/`.
- Notes are stored under `codex-notes/<task>/`.

## Error handling patterns

- Pages workflow currently fails fast on missing source directories during `cp -R`.
- `tsumeshogi-web-solver` app surfaces runtime errors via alerts / JSON text area.
- No shared site-level error handling exists for subprojects.

## Constraints

- This repo currently has no dedicated build step for third-party subprojects during Pages deploy; it only copies prepared assets.
- Bringing in `tsumeshogi-web-solver` can be done in two main ways:
  - vendor prebuilt static assets into this repo and copy them during Pages assembly
  - vendor source and extend this repo's Pages workflow to build the app during deploy
- If workflow is extended to build the app, the Pages job will need Rust toolchain, wasm target, Node install, and `wasm-pack`.
- Workflow edits must follow repo security rule: `actions/*` may use tags; any non-`actions/*` action must be pinned to a commit hash with an inline version comment.

## Potential pitfalls

- Copy-only integration is simple but requires checking in generated artifacts from another repo.
- Source-based integration is cleaner operationally but increases this repo's CI/deploy complexity and runtime.
- `tsumeshogi-web-solver` currently depends on npm install and Rust/WASM build tooling; lockfile / toolchain drift can break deploys.
- If upstream repo is copied in full, it brings its own `.github/workflows/` which should not be transplanted blindly into this repo.
- The app title link `../` is correct only if deployed at `/tsumeshogi-web-solver/` or deeper; it should be rechecked if path changes.

## Likely implementation directions

### Option A: import built static output

- Build `tsumeshogi-web-solver/www/dist`
- Commit it under a new directory in this repo
- Update `deploy-pages.yml` to copy it into `dist-pages/tsumeshogi-web-solver/`

Pros:

- Small change to site deploy workflow
- No Rust/wasm build in this repo's Pages workflow

Cons:

- Generated artifacts are checked in
- Upstream updates require rebuilding elsewhere or locally first

### Option B: import source and build in this repo's Pages workflow

- Add `tsumeshogi-web-solver/` source tree to this repo
- Extend Pages workflow to install Rust/Node dependencies and run the build
- Copy `tsumeshogi-web-solver/www/dist/.` into `dist-pages/tsumeshogi-web-solver/`

Pros:

- Source of truth stays in repo
- Reproducible build during deploy

Cons:

- More workflow changes
- More CI/deploy failure surface

## Recommendation

If the immediate goal is "publish this app under `koba-e964.com/tsumeshogi-web-solver/` with minimal repo-wide disruption", Option A is the lowest-risk integration because it matches the existing copy-based Pages assembly model already used by `fund-price-forecast/site/`.
